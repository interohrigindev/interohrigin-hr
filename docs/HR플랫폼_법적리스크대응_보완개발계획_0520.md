> 기준 문서: `docs/HR플랫폼_법적리스크대응_개발계획_0520.md`
> 보완일: 2026.05.20
> 목적: 법적 리스크 대응 기능을 "아이디어 문서" 수준에서 "실제 착수 가능한 실행 계획" 수준으로 재정렬

---

## 1. 보완 요약

원본 계획서는 방향이 매우 좋지만, 실제 저장소 규칙과 운영 제약을 반영하면 아래 5가지를 먼저 바로잡아야 한다.

1. `employees`, `evaluations`, `evaluation_items`, `users` 는 READ ONLY 테이블이므로 컬럼 추가나 구조 변경을 전제로 한 설계는 제외한다.
2. "핵심 테이블 audit 컬럼 추가" 대신 `audit_logs` 중심의 별도 감사 인프라로 설계한다.
3. 민감 데이터는 AI 판단 대상이 아니라 "사실 기록 + 제안" 중심으로 제한한다.
4. 모든 모듈은 일괄 오픈이 아니라 `feature flag + 관리자 수동 승인` 방식으로 단계 오픈한다.
5. `.bkit` 기준 현재 자동화 신뢰도는 높지 않다.
   - `automation.globalLevel = 2`
   - `trustScore = 40`
   - `team.enabled = true`
   - `hook reachability` 에서 `skill_post` 누락 기록 존재
   따라서 완전 자동 릴리스가 아니라 "작업 분리 + 검증 + 승인 게이트" 구조로 간다.

---

## 2. 이번 개발의 목표

### 비즈니스 목표

- 연장근로, 연차, 수습 종료, 징계/면담, 익명 신고, 법령 변경 대응에서 회사가 사전에 리스크를 줄일 수 있게 한다.
- 사건 발생 시 회사가 "언제, 누가, 어떤 승인/안내/통보를 했는지"를 재구성 가능한 형태로 남긴다.
- 인사팀이 수기 엑셀/메신저 기록에 의존하지 않도록 HR 플랫폼 안에서 증빙 체계를 일원화한다.

### 기술 목표

- READ ONLY 테이블은 건드리지 않고 신규 테이블 + RPC + 관리자 UI만으로 구현한다.
- 모든 민감 액션에 `audit_logs` 또는 등가의 별도 추적 로그를 남긴다.
- 관리자 화면은 모바일에서도 확인 가능해야 하며 날짜는 `YYYY.MM.DD` 형식을 사용한다.
- AI는 판단자가 아니라 "주의/권장/제안" 역할로만 쓴다.

---

## 3. 공통 설계 원칙

### 절대 준수

- `employees`, `evaluations`, `evaluation_items`, `users` ALTER 금지
- 한국어 UI, 날짜 `YYYY.MM.DD`
- 모바일 반응형 필수
- Phase 자동 전환 금지, 관리자 수동 승인
- AI 문구는 "결정"이 아닌 "제안/권장/점검 필요" 표현 사용
- 괴롭힘 신고, 징계, 연봉, 건강/가정 관련 메모는 임원/HR 외 비공개

### 아키텍처 원칙

1. 권한 검증은 `SECURITY DEFINER RPC` 우선
2. 민감 정보는 신규 테이블 분리 + 강한 RLS
3. 변경 이력은 기존 테이블 컬럼 추가 대신 `audit_logs` 에 집계
4. 실운영 오픈 전까지는 모듈별 feature flag 로 숨김 가능해야 함
5. PDF/메일/알림은 모두 재생성 가능해야 하며 발송 이력 저장

## 4. 우선순위 재정렬

## P0. 공통 기반

법적 리스크 대응 기능 전체의 기반이므로 제일 먼저 착수한다.

### P0-1. Audit Log 인프라

- 신규 테이블
  - `audit_logs`
  - `audit_exports`
- 저장 내용
  - actor uid
  - action type
  - entity type / entity id
  - before / after jsonb
  - request source
  - created_at
- 구현 원칙
  - READ ONLY 테이블은 직접 변경 감지 트리거를 강제하지 않는다.
  - 우선 신규 모듈과 관리자 액션부터 RPC 레벨에서 감사 로그를 남긴다.
  - 기존 수정 가능 테이블은 단계적으로 트리거 적용 여부를 검토한다.

### P0-2. 알림 디스패처 통합

- 기존 `/api/send-email` 확장
- 이메일 우선, 이후 Web Push, Slack/Webhook 확장
- 신규 테이블
  - `notification_deliveries`
  - `notification_templates`

### P0-3. 모듈 토글 + 운영 게이트

- 신규 테이블
  - `feature_rollouts`
  - `compliance_run_logs`
- 목적
  - 모듈별 시범 운영
  - 발송/경고/자동 생성 배치의 성공/실패 기록

## P1. 직접적인 분쟁 차단

### P1-1. 연장근로 사전 승인제

- 신규 테이블
  - `overtime_requests`
  - `overtime_actuals`
  - `overtime_policy_snapshots`
- 핵심 규칙
  - 사전 신청, 승인, 실제 종료 기록을 분리
  - 무승인 초과근무는 관리자 검토 필요 상태로 분류
  - 승인 종료 시각 대비 실제 종료 시각 편차 알림
- 화면
  - 직원 신청 화면
  - 승인 대기/반려 관리 화면
  - 초과근무 이력 타임라인

### P1-2. 주 52시간 사전 경고

- 기존 자산 재사용
  - `attendance_records`
  - `weekly_hours_tracking`
- 신규 테이블
  - `hours_warnings_sent`
  - `weekly_hours_snapshots`
- 핵심 규칙
  - 45h 노랑, 50h 빨강, 52h 초과 위험
  - 관리자와 직원에게 각각 다른 문구로 안내
  - 경고도 감사 대상

### P1-3. 연차 촉진 자동화

- 신규 테이블
  - `annual_leave_promotions`
  - `leave_balance_snapshots`
  - `leave_promotion_responses`
- 핵심 규칙
  - 소멸 6개월/2개월 전 자동 식별
  - 촉진 통지, 열람, 회신, 예정 사용일 입력을 전부 로그화
  - "수당 리스크 시뮬레이션"은 관리자 전용

## P2. 분쟁 발생 시 승소 가능성 강화

### P2-1. 징계/면담/통보 케이스 관리

- 신규 테이블
  - `disciplinary_cases`
  - `disciplinary_meetings`
  - `disciplinary_documents`
  - `disciplinary_notifications`
- 원칙
  - 징계 사유는 구조화 + 자유기술 혼합
  - 첨부 파일은 private storage + signed URL
  - 통보 경로와 수령/열람 상태를 별도 기록

### P2-2. 수습 종료 컴플라이언스 체크

- 원본 문서의 `employees.probation_decision_due_at` 제안은 제외한다.
- 대신 신규 테이블로 우회한다.
  - `probation_compliance_reviews`
  - `probation_alert_logs`
- 핵심 규칙
  - `hire_date + 90일` 및 사내 운영 기준으로 앱에서 계산
  - 30일 전 알림은 별도 배치에서 생성
  - 평가 객관성, 면담 횟수, 서면 통보 여부를 체크리스트로 저장

### P2-3. 익명 신고 핫라인

- 신규 테이블
  - `anonymous_reports`
  - `anonymous_report_replies`
  - `anonymous_report_files`
- 원칙
  - 제보자 실명 저장 금지
  - 토큰 기반 추적
  - 관리자 화면 접근 권한 최소화
  - "IP 무저장" 정책은 인프라 로그와 분리해 법무/보안 검토 후 확정

## P3. 운영 효율화

### P3-1. 법령 파라미터 자동 반영

- 신규 테이블
  - `legal_params`
  - `legal_param_fetch_logs`
  - `holiday_sync_logs`
- 원칙
  - 마지막 정상값 유지
  - 변경 감지만 자동화하고, 실제 운영 반영은 관리자 승인 후 적용 가능하게 설계

## 7. 완료 기준 (Definition of Done)

모듈별 완료는 아래 조건을 모두 만족할 때만 인정한다.

1. 신규 테이블/RPC/RLS 문서화 완료
2. 관리자/직원 화면 핵심 시나리오 동작
3. 모든 알림/배치 작업이 발송 또는 실패 로그를 남김
4. 민감 화면 접근 차단 확인
5. 날짜 형식 `YYYY.MM.DD`
6. 모바일 반응형 확인
7. `npx tsc -b` 또는 동등 빌드 통과
8. `npm run build` 통과
9. AI 문구가 판단형이 아니라 제안형으로 표시됨
10. READ ONLY 테이블 ALTER 없음

## 8. 운영 게이트

실운영 전 아래 게이트를 통과해야 한다.

### Gate A. 법무/노무 검토

- 촉진 통지 문구
- 연장근로 승인 문구
- 익명 신고 안내문
- 징계 통보 양식

### Gate B. 보안 검토

- 민감 파일 저장 위치
- signed URL 만료 정책
- 익명 신고 로그 정책
- 감사 로그 접근 권한

### Gate C. 운영 검토

- 모듈별 기본 OFF
- 시범 조직 지정
- 관리자 교육자료 준비

## 9. 주요 리스크와 보완책

| 리스크 | 보완 방향 |
|---|---|
| READ ONLY 테이블 변경 유혹 | 신규 테이블 우회 원칙 고정, 설계 리뷰 체크리스트 포함 |
| 감사 로그가 많아져 성능 저하 | 우선 신규 모듈 중심 로그부터 시작, 인덱스 설계, 아카이브 전략 마련 |
| 익명 신고가 실제 익명으로 인식되지 않음 | 저장 항목 공개, 법무 문구 검토, 토큰 기반 재접속 UX 명확화 |
| 자동 알림 과다 발송 | rollout 테이블로 부서/모듈별 단계 오픈, 발송 dry-run 지원 |
| AI가 민감 판단을 대신하는 오해 | UI 문구를 "권장/검토 필요"로 통일, 최종 결정자는 관리자 명시 |
| `.bkit` 훅 누락 상태로 자동화 과신 | `skill_post` 의존 없는 체크리스트 운영, 빌드/검증 결과는 수동 확인 기준 유지 |

---

## 10. 이번 문서 기준 확정 사항

1. READ ONLY 테이블은 절대 ALTER 하지 않는다.
2. `employees.probation_decision_due_at` 추가안은 신규 보조 테이블 방식으로 대체한다.
3. 감사 로그는 컬럼 추가형이 아니라 별도 로그형으로 구축한다.
4. 익명 신고는 MVP에서 토큰 기반 조회만 우선 지원한다.
5. 법령 패치는 "자동 감지 + 관리자 승인 적용"을 기본값으로 한다.
6. 에이전트 기반 개발은 메인 코디네이터 + DB + UI + QA 분리 구조를 권장한다.
