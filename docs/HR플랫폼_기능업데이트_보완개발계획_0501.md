# INTEROHRIGIN I&C — HR 플랫폼 & 복지몰 기능 업데이트 보완 개발 계획

> 기준 문서: `docs/HR플랫폼_기능업데이트_개발계획_0430.md`  
> 보완 작성일: 2026.05.01  
> 목적: 회의 요청사항을 실제 구현 가능한 개발 단위, 검증 기준, 선행 의존성, 단계별 오픈 기준까지 포함한 실행 계획으로 재정리

---

## 1. 계획 목적

- 원본 계획서의 요청사항을 기능 목록 수준에서 끝내지 않고, 실제 개발 착수 가능한 작업 묶음으로 전환
- HR 플랫폼과 복지몰 기능을 `운영 즉효성`, `오픈 필수성`, `외부 의존성`, `보안 영향도` 기준으로 재우선순위화
- 각 항목별로 `완료 기준(Definition of Done)`, `선행 조건`, `검증 방식`, `오픈 게이트`를 명확히 정의

## 2. 핵심 원칙

- 기존 READ ONLY 테이블 `employees`, `evaluations`, `evaluation_items`, `users`는 ALTER 금지
- 한국어 UI, 날짜 `YYYY.MM.DD`, 모바일 반응형 필수
- AI 출력은 `결정`이 아닌 `제안/권장` 표현 유지
- 전직원 오픈 전 보안 강화 완료 필수
- 기능은 한 번에 전사 배포하지 않고 `리더 → 운영팀 → 전직원` 순 단계 오픈
- 문서 작업이 아닌 실제 구현 단계에서는 수정 후 `npm run build` 검증 필수

## 3. 우선순위 재정렬

### P0. 오픈 필수
- 전자결재 웹/모바일 분리
- 지출결의서 양식 전면 개편
- 수습평가 회차 로직 강제
- 지원자 공유 링크/알림
- 전체 DB 보안 강화

### P1. 1차 오픈 효용 높음
- 프로젝트 관리 대시보드 재설계
- OJT 대상자 표시 오류 및 설정 UI 개선
- 정규직 평가 AI 가이드
- 헬스키퍼 예약 알림 자동화
- 자원 예약 시스템

### P2. 운영 보완
- 경조사 이미지 첨부
- 사무용품 요청 명칭/보류 정책 정리
- 전자서명 법적 효력 확인 이후 후속 구현

## 4. 모듈별 보완 계획

### 4.1 채용 / 지원자 관리

#### 목표
- 지원자 상세 페이지 하나로 채용 진행 현황, 면접 이력, 평가 흐름까지 파악 가능하도록 고도화

#### 개발 범위
- 지원자 상세 이력 확장
  - 면접 일시, 면접관, 면접 결과, 진행 단계, 단계별 코멘트 표시
- 외부 공유 링크
  - 로그인 없이 열람 가능한 읽기 전용 페이지
  - 만료일 또는 수동 비활성화 기능 검토
- 공고 정렬 로직 변경
  - 최신 지원자 접수 시간이 있는 공고 우선 정렬
- 지원자 접수 이메일 알림
  - 공고 담당자 기준 발송

#### 선행 조건
- 공유 토큰 또는 서명형 링크 전략 확정
- 채용 담당자 매핑 필드 확인

#### 완료 기준
- 지원자 상세 페이지에서 핵심 이력 누락 없음
- 공유 링크로 대표/외부 열람 가능
- 새 지원자 접수 시 담당자 메일 수신

### 4.2 전자결재 / 업무보고

#### 목표
- 모바일은 빠르게, 웹은 크게 작성할 수 있는 결재 UX로 분리
- 지출결의서가 실제 실무 결재 문서 수준으로 동작하도록 보강

#### 개발 범위
- 웹/모바일 분리
  - 모바일: 모달 유지
  - 웹: 전체 페이지 작성 방식
- 연장근무 시간 단위 30분 간격 변경
- 지출결의서 전면 개편
  - 실무 용어로 필드명 수정
  - 입금 계좌 정보 추가
  - 상세 내역 입력 영역 확대
  - Rich Text Editor 적용
  - 이미지 본문 삽입
  - 증빙 첨부 및 이미지 미리보기
- 업무보고 결재 양식에도 동일한 웹/모바일 분리 원칙 적용

#### 보류 항목
- 전자서명 법적 효력은 노무사 확인 전까지 설계만 유지

#### 완료 기준
- 웹에서 지출결의/업무보고 문서를 전체 화면으로 작성 가능
- 첨부 증빙을 결재자가 본문 또는 미리보기로 확인 가능
- 시간 선택지가 30분 단위로 정리됨

### 4.3 수습 / 정규직 평가

#### 목표
- 수습평가는 순서와 기간이 명확한 프로세스로 고정
- 정규직 평가는 작성 부담을 줄이고 평가자 판독성을 높임

#### 수습평가
- 회차 강조 UI
- D-day 임박 순 정렬
- 이전 회차 미완료 시 다음 회차 잠금
- 7일 활성화 후 자동 마감
- 관리자 강제 마감/잠금 해제

#### 정규직 평가
- 자기평가 AI 예시 가이드
  - 항목 설명 기반 정량/정성 목표 예시 생성
  - 예시 선택 후 부연설명 작성 구조
- 평가자 레이아웃 개선
  - 1행 3명 블럭형 배치

#### 완료 기준
- 잘못된 회차 평가 재현 불가
- 자기평가 작성 시 빈 화면이 아니라 예시 기반 입력 가능
- 평가자별 점수/코멘트 매칭이 쉬운 레이아웃 적용

### 4.4 프로젝트 관리

#### 목표
- 1단계 오픈 대상인 리더가 Monday.com 대체 수단으로 바로 사용할 수 있는 대시보드 완성

#### 개발 범위
- 참여인원별 기여도 그래프 복원
- 하단 `주의필요`, `담당자별 업무량` 블럭 재설계
- 테스트 데이터 정리
- 본인/참여 프로젝트 중심 노출 정리

#### 완료 기준
- 기여도 그래프 복원
- 하단 블럭에 대해 추가 설명 없이도 의미 이해 가능
- 리더 계정 기준 본인 관련 프로젝트만 노출

### 4.5 OJT / 교육 관리

#### 목표
- OJT 설정이 직관적으로 보이고, 대상자가 실제로 학습 내용을 확인할 수 있도록 안정화

#### 개발 범위
- 대상자 페이지 표시 오류 수정
- 설정 화면 개선
  - 실시간 프리뷰
  - 표 형식 스케줄 설정
  - 노션 수준의 직관적 구성 참고
- 모바일 동작 최종 점검

#### 완료 기준
- 설정 후 대상자 페이지에 OJT 내용이 정상 노출
- 일정표 입력이 표 형태로 가능
- 모바일 환경에서 핵심 기능 사용 가능

### 4.6 시스템 보안 강화

#### 목표
- CS 50만 건 이상 데이터가 있는 상태에서도 전직원 오픈 가능한 최소 보안 수준 확보

#### 개발 범위
- 전체 DB 보안 레이어 점검 및 강화
- 관리자 모니터링 시스템과 연계한 접근 점검
- 전직원 오픈 전 점검 체크리스트 문서화

#### 오픈 게이트
- 보안 강화 완료 전까지 전직원 오픈 금지

### 4.7 복지몰 결제 시스템

#### 목표
- PG 심사 전에는 포인트 기반 제한 오픈, 심사 후 정식 결제 오픈

#### 개발 범위
- 카카오페이 PG 심사 상태 추적
- 심사 전 임시 운영
  - 포인트 5만 한도 내 사용만 허용
- 심사 후 카드 결제/정산 정보 반영 구조 준비

#### 완료 기준
- 심사 전 운영 모드와 심사 후 정식 모드가 분리됨

### 4.8 헬스키퍼 예약

#### 개발 범위
- 예약 당일 푸시 알림 자동화
- 일반 계정 기준 예약 현황 노출 범위 확인

#### 완료 기준
- 당일 자동 알림 발송
- 타 직원 예약 노출 정책 명확화

### 4.9 자원 예약 관리

#### 목표
- 회의실, 스튜디오, 차량, 공용 노트북을 하나의 예약 체계로 통합

#### 개발 범위
- 시간 단위 타임슬롯 예약
- 자원별 중복 예약 차단
- 공용 노트북 대여/반납 상태 관리
- 공간 사진 및 직관적 이름 표시

#### 완료 기준
- 자원별 시간 충돌 예약 불가
- 공용 노트북 대여/반납 이력 추적 가능
- 회의실명이 사진과 함께 직관적으로 표시

### 4.10 사무용품 요청 / 경조사 / PWA

#### 사무용품 요청
- 기능명 `사무용품 요청`으로 통일
- 현재는 보류 상태 유지
- 인원 증가 시 활성화 기준 문서화

#### 경조사 관리
- 링크 만료 대비 이미지 첨부 기능 추가 검토

#### PWA
- 현재 구현 유지
- 향후 결제, 알림, QR 확인 흐름에 활용

## 7. 검증 기준

### 기능 검증
- 채용: 외부 공유 링크, 알림 메일, 최신 접수 기준 정렬 확인
- 결재: 웹/모바일 분리, RTE 입력, 첨부 미리보기 확인
- 평가: 회차 잠금/자동 마감/AI 예시 생성 확인
- 프로젝트: 본인 관련 항목 중심 노출, 그래프 복원 확인
- OJT: 대상자 노출, 스케줄 테이블, 프리뷰 확인
- 예약: 자원 충돌 차단, 알림 발송, 반납 상태 변경 확인

### 운영 검증
- 리더 계정/직원 계정/관리자 계정 각각 핵심 시나리오 점검
- 모바일과 웹에서 핵심 기능 진입 가능 여부 확인
- 메뉴 제한 오픈 설정 동작 확인

### 기술 검증
- `npm run build` 통과
- 보안 정책 및 권한 점검 완료
- 신규 테이블 또는 스토리지 정책이 READ ONLY 테이블 제약을 침범하지 않음
