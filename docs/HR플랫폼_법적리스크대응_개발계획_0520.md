# HR 플랫폼 — 법적 리스크 대응 모듈 개발 계획서

> 작성일: 2026.05.20
> 목적: 노무·근로기준법 위반 및 부당해고 분쟁 리스크에 대한 시스템 차원의 사전 차단·증빙 확보 기능을 IO HR 플랫폼에 통합

---

## Executive Summary

| 관점 | 내용 |
|---|---|
| **문제** | 인사 노무 영역의 법적 분쟁(부당해고·미지급 수당·52시간 위반·연차수당 청구) 발생 시 회사가 입증 자료를 시스템적으로 확보하지 못하면 패소·과태료·평판 손실로 직결됨 |
| **솔루션** | 기존 HR 플랫폼에 6개 컴플라이언스 모듈 추가 (연차 촉진/52시간 통제/법령 자동 패치/감사 이력/노무 진단/채용·퇴사 감사). 모든 작업·승인·변경에 위·변조 불가능한 audit trail 부여 |
| **범위** | DB: 신규 7~9개 테이블 + 핵심 테이블 audit 컬럼. UI: 8~10개 화면 (대시보드·신청·관리·익명 핫라인). 자동화: 4개 cron 작업 (촉진서·52시간 경고·법령 패치·수습 알림) |
| **핵심 가치** | ① 위반·분쟁 발생 자체를 시스템이 사전 차단 ② 발생 시 회사 증빙·승소율을 결정적으로 높임 ③ 인사 담당자의 수동 모니터링 부담 제거 |

> **결정 필요 사항**: 최종 실사용 여부는 모듈별로 별도 의사결정 가능. 본 계획서는 차개발 의뢰 기준 — 실 운영 전환은 모듈 단위 토글로 단계 적용 가능.

## 1. 현재 시스템 자산 vs 갭 매핑

| 신규 모듈 | 기존 자산 (재사용 가능) | 신규 필요 |
|---|---|---|
| **연차 자동화** | `leave_requests`, `employees.hire_date`, `/api/send-email` | 촉진서 템플릿·발송 이력·시뮬레이션 RPC |
| **주 52시간 통제** | `attendance_records`, `weekly_hours_tracking` | 임계치 자동 산정·실시간 경고·푸시/슬랙 연동 |
| **법령 실시간 패치** | `payroll_settings`, `company_events` | 정부 공공데이터 API 클라이언트·스케줄러 |
| **감사 이력 (AuditTrail)** | (없음 — 산발적 `updated_at` 만 존재) | 통합 audit_logs 테이블·트리거 기반 자동 캡처 |
| **노무 진단** | `bulletin_posts`(익명 글 가능), `approval_documents` | 익명 핫라인 + 연장근로 사전 승인 결재 양식 |
| **채용·퇴사 컴플라이언스** | `probation_evaluations`, `personnel_orders`, `exit_surveys` | 징계위 의결서·면담일지 양식·수습 만료 알림 cron |

**핵심 갭:** "감사 이력(audit_logs)" 인프라가 없음 → **모든 모듈의 공통 기반이므로 Phase 0 로 우선 구축 필요**

## 2. 단계별 우선순위 (법적 리스크 심각도 기준)

### Phase 0 — 공통 인프라 (1주)
모든 후속 모듈의 전제. 단독으로도 가치 있음.

- **M0-1. 통합 Audit Log 인프라**: `audit_logs` 테이블 + 핵심 테이블(employees·leave_requests·attendance_records·payroll·personnel_orders·probation_evaluations)에 BEFORE UPDATE/DELETE 트리거. 변경 전/후 jsonb + 변경자 uid + ip + user_agent 영구 보존. UI: 관리자 전용 "감사 로그" 화면 (검색·필터·CSV 내보내기).
- **M0-2. 알림 인프라 통합**: 이메일·푸시(브라우저 Web Push)·외부 메신저(Slack 옵션) 단일 디스패처. 기존 `/api/send-email` 확장.

### Phase 1 — 즉시 분쟁 차단 (2~3주, 법적 리스크 최고)
실제 손해배상·과태료가 가장 자주 발생하는 영역.

- **M1-1. 연장근로 사전 승인제** (소송 리스크 차단의 핵심)
- **M1-2. 주 52시간 사전 경고 대시보드**
- **M1-3. 연차 촉진 자동화 + 미사용 수당 시뮬레이터**

### Phase 2 — 분쟁 시 승소율 (2~3주)
사건 발생 시 회사 측 증빙을 결정짓는 영역.

- **M2-1. 채용·평가·징계 감사 (AuditTrail 확장)**
- **M2-2. 수습 만료 자동 알림 + 정당성 평가 템플릿**
- **M2-3. 직장 내 괴롭힘·성희롱 익명 신고 핫라인**

### Phase 3 — 운영 효율화 (1~2주)
법적 의무가 아니지만 자동화 효과 큼.

- **M3-1. 법령 실시간 패치 (최저임금·4대보험·공휴일)**

## 3. 모듈별 상세 설계

### M1-1. 연장근로 사전 승인제 (최우선)
**이유:** "내가 자발적으로 야근했다"는 사후 청구 소송이 가장 흔하고 패소율 높음.

| 항목 | 내용 |
|---|---|
| 신규 테이블 | `overtime_requests` (직원, 일자, 시작/종료 예정 시각, 사유, 승인자, 상태, 실제 종료 시각) |
| 신규 RPC | `request_overtime`, `approve_overtime`, `record_actual_overtime` |
| UI 1 | 직원: 야근 사전 신청 폼 (사유 필수, 결재 라인) |
| UI 2 | 관리자: 승인 대기 리스트 + 일괄 승인 |
| 비즈 룰 | 사전 미승인 시 attendance_records 에 추가 시간이 들어와도 "비인정" 표시. 실제 종료 시각 vs 승인 시각 차이 30분 초과 시 별도 보고. |
| 증빙 | 모든 신청·승인·실제 기록에 audit_logs 자동 기록 |

### M1-2. 주 52시간 사전 경고
| 항목 | 내용 |
|---|---|
| 신규 테이블 | 없음 — `weekly_hours_tracking` 활용 |
| 신규 RPC | `compute_weekly_hours_warning(p_employee_id, p_week_start)` |
| UI 1 | 직원 홈 위젯: "이번 주 누적 시간 / 52시간 진행도 바" |
| UI 2 | 관리자 대시보드: 부서별·직원별 위험도 매트릭스 (45h 노랑/50h 빨강/52h 위반) |
| 자동화 | 매일 06:00 cron → 전 직원 주간 시간 계산 → 임계치 도달자에게 이메일/푸시 발송 + 관리자 알림 |
| 증빙 | 경고 발송 이력 별도 테이블 (`hours_warnings_sent`) |

### M1-3. 연차 자동화
| 항목 | 내용 |
|---|---|
| 신규 테이블 | `annual_leave_promotions` (직원, 발송일, 회차 6/2개월, 미사용 일수, 응답 일자, 응답 내용 jsonb), `leave_balance_snapshots` (직원, 기준일, 발생/사용/잔여, 예상 수당액) |
| 신규 RPC | `compute_leave_balance(p_employee_id)`, `send_leave_promotion(p_employee_id, p_stage)`, `simulate_unused_leave_liability()` |
| 자동화 | 매일 09:00 cron → 소멸 6/2개월 전 도달자 식별 → 촉진서 자동 발송 + 사용 예정일 입력 요청 |
| UI 1 | 직원: 촉진서 수신 알림 + 사용 예정일 입력 화면 (열람 로그 자동) |
| UI 2 | 관리자: 회사 전체 미사용 수당 잠재 부채 대시보드 (월별 추이 차트) |
| 법적 효과 | 촉진 절차 완료 시 수당 지급 의무 소멸 — 시스템 로그가 그 증빙 |

### M2-1. 채용·평가·징계 감사 확장
| 항목 | 내용 |
|---|---|
| 신규 테이블 | `disciplinary_actions` (대상자, 사유 jsonb, 위원회 의결서 URL, 면담 일지 jsonb 배열, 결정, 결정일, 통보 방식·일자, audit) |
| 기존 활용 | `personnel_orders` 와 연결 (대기발령·부서이동) |
| UI 1 | 관리자: 징계 케이스 등록 → 단계별 양식 (사실 확인서 → 위원회 의결 → 통보 → 결과) |
| UI 2 | 케이스별 타임라인 + PDF 출력 (구제신청 대응용) |
| 증빙 | 모든 변경 audit_logs + 파일 첨부 (S3 immutable bucket 권장) |

### M2-2. 수습 만료 자동 알림
| 항목 | 내용 |
|---|---|
| 기존 활용 | `probation_evaluations`, `employees.hire_date`, `employees.probation_result` |
| 신규 컬럼 | `employees.probation_decision_due_at` (수습 만료 30일 전 자동 계산값) |
| 신규 RPC | `send_probation_expiry_alert(p_employee_id)` |
| 자동화 | 매일 cron → 30일 전 도달 직원 식별 → 결재권자(임원/대표)에게 알림 + 평가 미완료 항목 강조 |
| UI | 기존 수습평가 화면에 "법정 정당성 체크리스트" 섹션 추가 (서면 통보 의무·평가 객관성·면담 횟수 등 자동 점검) |

### M2-3. 익명 신고 핫라인
| 항목 | 내용 |
|---|---|
| 신규 테이블 | `anonymous_reports` (제보자 익명 토큰, 카테고리(괴롭힘/성희롱/기타), 내용, 첨부, 접수일, 처리 상태, 처리 이력 jsonb) — 제보자 실명 절대 저장 X |
| RLS 정책 | INSERT: anon 가능 (토큰 발급). SELECT: 인사팀·법무 담당자만 (별도 role) |
| UI 1 | 공개 페이지 `/report-anonymous` (로그인 불필요, IP 로그도 안 함) |
| UI 2 | 인사팀 전용 처리 화면 (응답은 토큰 기반 답변 페이지) |
| 보안 | 작성 후 토큰 발급 → 제보자만 그 토큰으로 추가 응답·확인 가능 |

### M3-1. 법령 실시간 패치
| 항목 | 내용 |
|---|---|
| 데이터 소스 | 공공데이터포털 API (최저임금, 4대보험 요율), 인사혁신처 공휴일 API |
| 신규 테이블 | `legal_params` (key, value jsonb, effective_from, source, fetched_at) |
| 자동화 | 매주 cron → API 호출 → 변경 감지 → 저장 + 관리자 알림 |
| 기존 활용 | `payroll_settings`, `company_events` 가 `legal_params` 참조하도록 점진 마이그레이션 |

## 4. 공통 설계 원칙 (절대 규칙 준수)

- **기존 ALTER 금지 테이블 유지**: `employees`, `evaluations`, `evaluation_items`, `users` 는 컬럼 추가 없이 신규 테이블에 FK 로만 연결
  - 단 `employees.probation_decision_due_at` 의 경우 별도 view 또는 `employee_hr_details` 활용으로 우회 검토
- **권한 검증은 SECURITY DEFINER RPC**: 098 마이그레이션 패턴 재사용 — RLS 표현식 의존 최소화 (반복 차단 사고 방지)
- **모든 destructive UI 액션 confirm 다이얼로그 + audit 기록**
- **한국어 UI, 날짜 YYYY.MM.DD, 모바일 반응형**
- **AI 사용 시 "결정 X / 제안 O" 표현**
- **민감 정보(괴롭힘 신고·징계 사유·연봉)는 임원/HR 외 비공개 RLS**

## 5. 의사 결정 필요 항목

본 계획서는 차개발 의뢰 기준이며, 실 운영 전 다음 사항 확정 필요:

1. **외부 메신저 연동 범위**: Slack 만 / Slack + Teams / Webhook 만
2. **익명 핫라인 응답 권한**: 인사팀장 단독 vs 임원 위원회 합의 후 응답
3. **연장근로 승인자 라인**: 직속 리더 → 임원 vs 즉시 임원
4. **공공데이터 API 사용**: 인증키 발급 주체 (회사 명의)
5. **징계 의결서 파일 저장소**: 기존 Supabase Storage vs 별도 immutable 버킷
6. **시범 운영 모듈 선정**: Phase 1 만 먼저 / 전체 동시 오픈
7. **법무 검토 단계 포함 여부**: 운영 전 외부 노무사 감수 1회 권장

## 7. 리스크

| 리스크 | 완화 방안 |
|---|---|
| 익명성 보장 실패 시 더 큰 분쟁 | DB 컬럼·로그·트리거 모두 제보자 ID 비저장 점검, 외부 보안 감수 |
| 정부 API 장애로 법령 패치 실패 | 마지막 성공 값 유지 + 관리자 수동 패치 옵션 |
| 직원이 사전 승인 안 받고 야근 강행 | 비인정 처리 + 출퇴근 기록과 자동 대조 알림 |
| 감사 트리거 성능 저하 | 핵심 테이블만 우선 적용, 인덱스 + 파티셔닝 |
| 기존 데이터 누락된 audit 이력 | "이력 시작일" 명시 + 그 이전은 별도 백업 참조 |
