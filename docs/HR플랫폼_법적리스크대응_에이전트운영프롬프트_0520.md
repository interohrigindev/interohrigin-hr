# HR 플랫폼 — 법적 리스크 대응 에이전트 운영 프롬프트

> 기준일: 2026.05.20
> 기반 컨텍스트: `.bkit/*`, `docs/HR플랫폼_법적리스크대응_보완개발계획_0520.md`
> 목적: `.bkit` 상태를 반영해 메인 에이전트와 작업 에이전트가 병렬로 안전하게 개발할 수 있는 운영용 프롬프트 제공

---

## 1. `.bkit` 컨텍스트 요약

현재 저장소의 `.bkit` 상태에서 읽힌 핵심 정보는 아래와 같다.

- `team.enabled = true`
- `pipeline.currentPhase = 1`
- `pipeline.level = Dynamic`
- `automation.globalLevel = 2`
- `automation.trustScore = 40`
- `activeFeatures = []`
- `hook reachability` 에서 `skill_post` 누락 기록 존재
- `audit/*.jsonl` 에 명령/수정 이력이 기록되고 있음

### 이 의미

1. 에이전트 병렬 작업은 가능하지만 완전 자동 위임형 운영에는 아직 보수적으로 접근해야 한다.
2. 각 에이전트는 작업 범위를 분리하고, 결과를 메인 에이전트가 통합 검증해야 한다.
3. 훅 누락 이력이 있으므로 "훅이 알아서 완료를 잡아주겠지" 전제를 두면 안 된다.
4. 감사 로그가 강하게 남으므로 명령은 단순하고 추적 가능하게 유지하는 것이 좋다.

---

## 2. `.bkit` 기반 운영 원칙

### 권장

- 메인 코디네이터 1명 + 작업 에이전트 3명까지
- 파일 소유권을 분리
- 각 작업 종료 시 변경 파일 목록과 검증 결과를 텍스트로 보고
- 빌드/최종 통합은 메인 에이전트가 수행

### 주의

- `rm`, 광범위한 치환, 복합 heredoc, `$()` 내부 heredoc 등은 피한다.
- 단일 명령, 짧은 파이프라인, 명확한 파일 경로를 사용한다.
- 감사 로그 및 훅 상태상 "보이지 않는 자동 처리"에 의존하지 않는다.

## 3. 권장 에이전트 구성

## Agent 0. Main Coordinator

### 책임

- 문서/규칙 확인
- 작업 범위 분할
- 각 에이전트 결과 통합
- 충돌 조정
- 최종 빌드 및 오픈 판단

### 소유 범위

- `docs/*`
- 최종 통합 파일
- `src/components/layout/Sidebar.tsx`
- 공통 설정/검증

## Agent 1. DB / RPC Agent

### 책임

- 신규 테이블
- RLS
- RPC
- 타입 갱신

### 소유 범위

- `supabase/migrations/*`
- `src/types/database.ts`
- `src/lib/*audit*`
- `src/lib/*notification*`

## Agent 2. UI / Flow Agent

### 책임

- 관리자/직원 화면
- 폼 흐름
- 반응형 정리

### 소유 범위

- `src/routes/attendance/*`
- `src/routes/leave/*`
- `src/routes/ojt/*`
- `src/routes/employees/*`
- `src/routes/report-anonymous.tsx`

## Agent 3. QA / Docs Agent

### 책임

- 시나리오 점검
- 문구 정책 확인
- 오픈 체크리스트 정리

### 소유 범위

- `docs/*`
- 테스트 리포트
- 운영 가이드

## 4. 메인 코디네이터 시작 프롬프트

```text
당신은 INTEROHRIGIN HR Platform의 메인 코디네이터 에이전트다.

작업 목표:
- 법적 리스크 대응 모듈을 보완 개발 계획서 기준으로 구현한다.

반드시 지킬 규칙:
- employees, evaluations, evaluation_items, users ALTER 금지
- 기존 패턴/네이밍/색상 유지
- 한국어 UI, 날짜 YYYY.MM.DD, 모바일 반응형
- AI 문구는 제안/권장 표현
- 민감 정보는 임원/HR 외 비공개
- 자동화 신뢰도가 높지 않으므로 모든 위험 작업은 수동 검토 게이트 포함

.bkit 컨텍스트:
- team.enabled = true
- globalLevel = 2
- trustScore = 40
- hook reachability 에 skill_post 누락 이력 존재
- audit 로그가 강하게 남으므로 명령은 단순하게 유지

당신의 역할:
1. 작업을 DB/RPC, UI/Flow, QA/Docs 로 분리
2. 각 에이전트의 소유 파일을 겹치지 않게 지정
3. 먼저 P0부터 착수
4. 결과를 통합한 뒤 npx tsc -b 와 npm run build 검증
5. 마지막에 변경 파일, 검증 결과, 남은 리스크를 정리
```

## 8. 병렬 운영 순서

1. Main Coordinator 가 계획서/규칙/소유 범위를 확정
2. DB / RPC Agent 가 P0 구조를 먼저 정의
3. UI / Flow Agent 는 DB 계약이 필요한 부분만 대기하고, 독립 가능한 화면부터 선작업
4. QA / Docs Agent 는 병렬로 체크리스트와 테스트 케이스 작성
5. Main Coordinator 가 통합 후 빌드/리뷰/최종 보고

## 9. 메인 에이전트 최종 통합 프롬프트

```text
지금부터는 메인 코디네이터로서 각 에이전트 산출물을 통합해줘.

체크리스트:
- READ ONLY 테이블 ALTER 없음
- 신규 테이블/RPC/RLS 연결 확인
- UI 가 한국어/모바일/날짜 형식을 준수하는지 확인
- AI 문구가 제안형인지 확인
- 민감 정보 접근 차단 확인
- 감사 로그 / 알림 로그 / 배치 로그가 모두 추적 가능한지 확인
- npx tsc -b
- npm run build

최종 보고 형식:
1. 구현 완료 항목
2. 변경 파일
3. 검증 결과
4. 운영 전 남은 의사결정
5. 추천 오픈 순서
```

## 10. 권장 실행 순서

가장 안정적인 시작 순서는 아래와 같다.

1. `보완개발계획` 승인
2. `실행프롬프트`의 P0 실행
3. `에이전트운영프롬프트`로 DB/UI/QA 병렬 착수
4. P1 완료 후 시범 오픈
5. P2, P3 는 운영 피드백 반영 후 확장
